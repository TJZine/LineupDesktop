## Points Of Contention

- The largest implementation risk is not the PIN path; it is selected-server
  persistence and startup routing. Upstream scopes selected server and resume
  policy by active user/profile, while Desktop currently stores a renderer-safe
  selected-server summary without raw connection details.
- Discovery request policy is a likely cause of current live server failures.
  Desktop needs upstream-equivalent variants, XML fallback, and retry/rate-limit
  behavior before more live patching.
- Plex Home endpoints and protected switch behavior need a deliberate rewrite
  rather than incremental patches because endpoint version, PIN placement, and
  fallback policy are coupled.
- The renderer should not preserve the existing setup/debug control panel as a
  product surface. The replacement should copy upstream product flow semantics,
  not necessarily upstream DOM structure.
- Library section parsing is mostly aligned, but browse/search/metadata behavior
  has targeted parity gaps that should be fixed with tests before live proof.
- `types` support for search and filter support for browse may require a narrow
  contract/preload unit. That unit should remain optional until implementation
  proves the current public vocabulary cannot express the reviewed behavior.
- Adjacent fake player/guide/settings surfaces remain outside Unit 1 and cannot
  be used as readiness proof for RD-22.
